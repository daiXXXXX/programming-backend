# 本地样例测试功能说明

## 需求背景

用户在正式提交代码之前，通常希望先基于题目样例验证当前实现是否符合预期。原有流程只有“提交评测”，会直接写入提交记录，不利于用户先做轻量验证。

本次新增“本地样例测试”能力，使用户可以在正式提交前先运行公开样例，并查看每个测试点的输入、期望输出、实际输出/错误信息和耗时。

## 涉及文件

### 后端

- `internal/models/models.go`
- `internal/handlers/submission.go`
- `cmd/server/main.go`

### 前端

- `src/lib/api.ts`
- `src/components/ProblemDetail.tsx`
- `src/lib/i18n/zh.ts`
- `src/lib/i18n/en.ts`

## 具体改动

### 1. 后端新增样例测试接口

新增接口：`POST /api/submissions/test`

用途：
- 接收题目 ID、代码、语言
- 仅运行公开样例测试
- 不写入数据库，不生成正式提交记录
- 返回运行状态、得分、测试结果详情、运行时间

### 2. 后端新增本地测试数据模型

在 `models.go` 中新增：
- `RunSampleTestsRequest`
- `CodeRunResult`

用于统一描述本地样例测试的请求与返回结构。

### 3. 后端样例选择逻辑

在 `submission.go` 中新增样例筛选逻辑：
- 优先运行 `is_sample = true` 的测试用例
- 最多运行 3 个公开样例
- 若题目没有显式标记样例，则回退到前几个测试用例，保证功能可用

这样既满足“提交前先测样例”的需求，也避免把完整判题数据暴露给本地测试。

### 4. 前端新增样例测试 API

在 `src/lib/api.ts` 中新增：
- `runSampleTests()`
- `CodeRunResult` 类型

前端可直接调用新接口完成样例测试。

### 5. 题目详情页新增“本地测试”按钮

在 `src/components/ProblemDetail.tsx` 中：
- 新增“本地测试”按钮
- 与“提交”按钮并排展示
- 本地测试执行期间展示加载状态
- 切换题目时清理上一题的测试结果，避免串题

### 6. 新增本地测试结果展示区域

题目详情页中新增独立结果卡片，用于展示样例测试结果：
- 当前状态
- 得分
- 每个测试点的输入
- 期望输出
- 实际输出或错误信息
- 执行耗时

同时对正式提交后的结果卡片也复用了同一套详情展示逻辑，提升信息完整性。

### 7. 国际化文案补充

在中英文文案中新增以下内容：
- 本地测试按钮文案
- 测试中状态
- 样例测试结果标题
- 功能提示文案
- 测试完成/失败提示
- 错误信息文案
- 无测试结果文案

## 实现原因

本次改动的核心目标是把“轻量验证”和“正式提交”区分开：

1. **减少无效提交**
   - 用户可以先跑样例，减少明显错误直接进入正式评测。

2. **提升交互体验**
   - 不需要每次都生成提交记录即可看到测试反馈。

3. **保护判题数据**
   - 本地测试只运行公开样例，不直接暴露完整测试集。

4. **便于调试**
   - 明确展示输入、期望输出、实际输出、错误信息和耗时，方便用户快速定位问题。

## 当前效果

用户现在可以在题目详情页中：
- 先点击“本地测试”运行样例
- 查看详细的样例测试结果
- 确认无误后再点击“提交”进入正式评测流程

这样就形成了更完整的“编写代码 → 样例验证 → 正式提交”的使用链路。
